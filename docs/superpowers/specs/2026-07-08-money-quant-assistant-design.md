# money — 量化分析助手（CLI + Claude Skill）设计文档

- 日期：2026-07-08
- 状态：设计已确认，待评审
- 语言/栈：Python

## 1. 定位（一句话）

Python 负责**取数 + 算指标 + 收集舆情原始信息 + 存档复盘**，把结果打包成一张带版本的**「证据卡」**；Claude（通过 Claude Code Skill）读证据卡做**综合推理**，产出**信号 + 理由 + 风险 + 置信度**。不做黑箱 ML，判断可解释、可追溯、可复盘。

## 2. 目标 / 非目标

**目标**
- 覆盖两类资产：**场外基金（开放式基金）** 与 **黄金**（上海金交所 Au99.99 / 黄金 ETF）
- 任意代码/名称输入即可分析
- 由 Claude 综合"量化证据 + 舆情证据"给出可解释的买/观望/回避信号
- **可度量、可复盘**：每次信号存档，事后比对真实涨跌算命中率，Claude 分析前先看自己历史战绩以校准
- 结构上为"推广"预留：CLI 可 `pip install`、Skill 可拷贝即用、判断框架是公开可读的资产

**非目标（YAGNI，明确不做）**
- ❌ 不做精确点位/精确收益预测（数学上不可能，任何此类承诺都是过拟合或欺骗）
- ❌ 不训练自建情感/价格 ML 模型（Claude 本身即"判断引擎"）
- ❌ MVP 不做网页看板、不做无人值守定时（放 P2/P3）
- ❌ 不写死个人持仓清单

**诚实边界**：本工具是"基于证据的理性参谋"，帮助减少拍脑袋决策，**不保证盈利**，最终决策与风险由使用者承担。

## 3. 核心架构：一条缝 + 两个契约

### 3.1 一条缝 = 证据卡（确定层 / 判断层分离）

```
────────────── 判断层（Skill = Claude）──────────────
  编排 · WebSearch 抓舆情 · 鉴别真伪 · 综合推理 · 出信号+理由+风险+置信度
        ▲ 只依赖 证据卡 schema，不碰 CLI 内部实现
════════ 缝 = evidence_card JSON（带 schema 版本）════════
        ▼ 只产出客观数据，永不下结论
────────────── 确定层（CLI = Python）──────────────
  取数 · 算指标 · 算历史分位 · 收集舆情原始信息 · 存档信号 · 复盘战绩
```

**切缝原则**：凡是**能复现、可单测、别人能独立验证**的（指标算得对不对、数据取得全不全）放 CLI；凡是**需要判断、带主观性**的（这条新闻可不可信、该不该买）放 Skill。数据源或指标实现更换时，只要证据卡 schema 不变，Skill 一行不改；反之亦然。

### 3.2 两个共享契约

1. **证据卡 schema**（`evidence_card`，pydantic + 版本号）：CLI 的产出、Skill 的唯一输入。
2. **分析框架**（`quantfox/prompts/analysis_framework.md`）：Claude "怎么判断"的**唯一真理源**——信号档位定义、必须说明采信/忽略了哪些舆情、必须给风险与置信度。SKILL.md **引用**它，不另写一套。

**为什么要两个契约**：今天是 Claude 在 CC 交互里当大脑（无需 API key、不花钱）。将来 P2 做无人值守每日定时/对外产品，需要 headless 跑法（`quantfox analyze --llm`，走 Claude API）。届时它消费**同一张证据卡 + 同一份分析框架**——两个运行器，一个大脑规范，绝不分叉。今天把框架抽成独立文件，明天零重构接上。

## 4. 组件（各自单一职责）

```
money/
  pyproject.toml
  money/
    cli.py                 # typer 命令行入口
    data/
      funds.py             # 场外基金历史净值（akshare 封装）
      gold.py              # 黄金 Au99.99 / 黄金 ETF（akshare 封装）
      news.py              # 舆情原始信息收集（akshare 新闻接口，只收集不预判）
      resolve.py           # 代码/名称解析、资产类型判定
    indicators.py          # 技术指标（pandas-ta 封装）
    percentile.py          # 历史分位（point-in-time，防前视偏差）
    evidence.py            # 组装证据卡（pydantic 模型 + 版本）
    storage.py             # 信号存档 + 复盘（SQLite）
    prompts/
      analysis_framework.md  # 分析框架（判断层唯一真理源）
  .claude/skills/fund-analyze/SKILL.md
  tests/                   # 单测，用离线 fixtures，不打实时网络
  docs/
```

**我们真正手写的只有 4 样薄东西**：证据卡组装器（`evidence.py`）、分析框架（`analysis_framework.md`）、Skill（`SKILL.md`）、信号存档（`storage.py`）。其余全部调成熟库。

## 6. CLI 子命令（细粒度、可组合）

| 命令 | 作用 | 谁调 |
|---|---|---|
| `quantfox evidence <代码/名> [--format json\|markdown]` | 产出完整证据卡（主命令） | Skill 主流程 |
| `quantfox fetch <代码/名>` | 只取原始净值/价格 | 调试/组合 |
| `quantfox indicators <代码/名>` | 只算技术指标 | 调试/组合 |
| `quantfox news <代码/名>` | 只收集舆情原始信息 | 调试/组合 |
| `quantfox log-signal --symbol X --signal buy --confidence 0.6 ...` | 把本次信号 + 置信度 + 当时证据快照存进预测账本 | Skill 出结论后 |
| `quantfox outcomes [--recompute]` | 为到期的历史预测算真实收益（可复算） | 定期/复盘前 |
| `quantfox review <代码/名>` | 单标的战绩：命中率、超额收益 | Skill 分析前先看战绩 |
| `quantfox review --all [--since <版本>]` | 全局战绩：IC、命中率、置信度校准、vs 基准、按大脑版本分组 | 评估/推广展示 |

存储由 CLI 独占（确定、可测）；Skill 只负责"触发写入"与"读战绩来校准"。

## 7. 证据卡 schema（字段草案）

```jsonc
{
  "schema_version": "1.0",
  "asset": { "symbol": "...", "name": "...", "type": "otc_fund|gold", "as_of": "YYYY-MM-DD" },
  "price": {
    "latest": 1.234, "latest_date": "YYYY-MM-DD",
    "return": { "1w": 0.01, "1m": -0.02, "3m": 0.05, "1y": 0.12 },
    "max_drawdown_1y": -0.18, "volatility_1y": 0.15
  },
  "indicators": {
    "ma": { "ma5": ..., "ma10": ..., "ma20": ..., "ma60": ..., "alignment": "多头|空头|纠缠" },
    "macd": { "dif": ..., "dea": ..., "hist": ..., "state": "金叉|死叉|—" },
    "rsi14": ..., "boll": { "pos": "上轨|中轨|下轨附近", "width": ... }
  },
  "percentile": { "price_pct_3y": 0.22, "note": "近3年价格分位，point-in-time" },
  "news": [
    { "title": "...", "source": "...", "date": "...", "url": "...", "summary": "..." }
  ],
  "track_record": { "past_signals": N, "hit_rate": 0.55, "ic": 0.08, "vs_benchmark": 0.02, "note": "该标的历史信号：命中率/信息系数/相对基准超额，供 Claude 校准" },
  "data_quality": {
    "price": "ok|stale|partial|missing",
    "news": "ok|sparse|missing",
    "notes": ["净值更新至 T-1，正常" ]
  }
}
```

数据缺失一律显式 `null` + `data_quality` 说明，**绝不静默补空**。

## 8. 分析框架（`analysis_framework.md` 要点）

- **信号档位**：强买 / 买 / 观望 / 减 / 回避 —— 每档给出量化+舆情的判定倾向（非硬阈值，作为 Claude 推理的参照）
- **输出必须包含**：
  1. 信号 + 置信度（并根据 `data_quality` 缺失情况下调置信度）
  2. 采信了哪几条舆情、为什么；忽略了哪些噪音/营销，为什么
  3. 量化证据如何支撑或反驳
  4. **风险与不确定性**（必写）
  5. 参照过去战绩（`track_record`）做的校准说明
- **禁止**：给精确点位/精确收益承诺；把 `null` 当依据；无理由的乐观

## 9. 标准量化的严谨性（玩具 vs 专业的分水岭）

- **无前视偏差（no look-ahead）**：历史分位、指标一律 point-in-time 计算，只用截至当日可得的数据。
- **复盘用真实持有期收益**：`review` 比对信号后 N 日/持有期的真实净值变化，而非拟合曲线。
- **场外基金 T+1 特性**：净值为每日一个、当日盘中不可得，信号面向"次日申购"语义，复盘据此对齐。
- **防过拟合**：不堆砌大量参数/指标去"凑对历史"；指标集保持精简、含义清晰。
- **交易成本意识**：复盘/未来回测计入申赎费、持有时长限制（场外基金短期赎回费高）。
- **样本外意识**：战绩统计区分"近期 vs 全程"，避免只看好看的区间。

## 9A. 预测账本与复盘评估（"越用越准"的科学核心）

这是全项目最重要的子系统：让"尽量准"可度量、让"越用越准"可验证、让"推广"有真实战绩可亮。

### 9A.1 预测账本（append-only，不可篡改）

表 `predictions`，每次分析冻结一条，**永不修改**（改历史预测 = 作弊，直接毁掉可信度）：

- `id`、预测日期 `T0`、标的（symbol/name/type）
- 信号（强买…回避）+ **数值化**（如 -2…+2）、置信度（0–1）
- **评估周期 horizons**：5 / 20 / 60 交易日 —— *无周期则无法算账，必填*
- `T0` 参考价格/净值
- **当时证据卡完整快照（JSON）** —— 供日后回归"到底哪些证据真有预测力"
- Claude 的推理理由 + 采信/忽略了哪些舆情
- **大脑版本**：`analysis_framework` 版本号 + `schema_version`

### 9A.2 到期算账（outcome，可复算、point-in-time）

outcome = `预测 + 之后真实净值` 的**纯函数**，可随时重算（因此无法作弊）：

- 各周期真实收益（T0→T0+h）、真实最大回撤（看路径，不只看终点）
- **基准对比**：vs 买入持有（benchmark）、vs 对应指数 → *"信号比躺着不动到底强没强"*（最诚实的标尺）
- 是否判对（方向 / 分档打分）

### 9A.3 复盘指标（用户要的"回归分析：预测准不准"）

| 人话 | 专业指标 | 意义 |
|---|---|---|
| 方向对的比例 | 命中率（hit rate） | 最直观 |
| 信号强弱 vs 实际涨跌的相关性 | **IC（信息系数）** | 量化行业衡量"预测力"的标准指标 |
| 说 70% 把握时是否真约 70% 对 | **置信度校准（calibration）** | "尽量准"的诚实度 |
| 比躺着持有多赚/少亏 | **超额收益** | 有没有真正创造价值 |

按 标的 / 周期 / 市场环境 / 大脑版本 分组统计。

### 9A.4 防自欺（专业严谨）

- **point-in-time**：outcome 只用 T0 之后的数据。
- **append-only**：历史预测永不改写。
- **全量统计，不 cherry-pick**：报告所有预测，不挑好看的。
- **必对基准**：牛市里啥都涨，不比 benchmark 就是自欺。
- **样本不足直说**：N 太小时明确"数据不够、别当真"，给置信区间。
- **样本外**：区分"改框架前 vs 改框架后"分开统计。

### 9A.5 改进闭环（"方便后续改进"）

- 每次分析前，Claude 先读该标的/该环境的历史战绩 → **校准**（例："历史上我对黄金 RSI<30 的买入只对 45%，这次下调把握"）。
- 改 `analysis_framework.md` → 版本号 +1 → `review --since <版本>` 对比新旧版本准不准 = **给自己做 A/B**，用数据证明改进有没有用。
- 于是"预测 → 算账 → 回归 → 改进 → 再验证"形成闭环飞轮。

## 10. 数据流（主链路）

```
/fund-analyze 501018
  → Skill 读 SKILL.md + analysis_framework.md
  → Bash: money review 501018        （先看战绩）
  → Bash: money evidence 501018       （取证据卡 JSON）
  → WebSearch/WebFetch：补充最新舆情
  → Claude 综合推理（按分析框架）
  → 输出：信号 + 理由 + 采信/忽略的舆情 + 风险 + 置信度
  → Bash: money log-signal ...        （存档本次信号）
```

## 11. 错误与数据质量处理

- 代码/名称解析失败或多义 → CLI 返回候选列表，Skill 与用户澄清。
- 数据源超时/接口变动 → CLI 明确报错 + `data_quality=missing`，不伪造。
- 网络抖动 → 数据层单测用离线 fixtures，不依赖实时网络。

## 12. 测试策略

- **数据层**：离线保存样本（akshare 返回快照）测解析，不打实时网络。
- **指标层**：TDD，已知输入→已知输出验证 MACD/RSI/分位计算正确。
- **证据卡**：pydantic 校验 + 快照测试，验证结构完整、缺失字段正确标 `null`。
- **存档/复盘**：用构造数据验证命中率/收益计算正确。

## 13. 分期

- **P1 / MVP（现在）**：数据（基金+黄金）+ 指标 + 历史分位 + 舆情收集 + 证据卡 + 信号存档/复盘 + Skill（Claude 分析）。
- **P2**：每日 `loop` 定时（headless `--llm` 走 Claude API）、回测（vectorbt/backtrader）、更多结构化数据源。
- **P3**：网页 K 线看板、正式对外发布与文档。
