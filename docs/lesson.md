# 操作级教训（不是用户档案）

每条：场景 / 我做错的 / 下次怎么避免。

## 2026-07-10 · "深度筛选基金"被理解成"看收益榜前几名"

- **场景**：用户要"深度思考筛选十分靠谱、高收益、高概率正收益"的基金。
- **我做错的**：直接跑 `quantfox screen --consistent` 拿了各类型**前12名**就当筛选结果。而 `screen` 的打分本质是"多周期收益加权排名"（见 screen.py），榜首必然是近期涨最猛的过热单一板块（全是半导体/AI，近3月翻倍、站52周最高点）。这不是筛选，是看榜；且全挤在一个主题，毫无分散。用户明确不满："没按要求深度筛选，找了已有的、不合适的。"
- **用户点破的本质**：找基金是为赚钱，**希望我把全市场尽量都看一遍**（全面），而不是端现成榜单。
- **下次怎么避免**：
  1. "深度/全面筛选" = 拉**全市场全量**（load_universe 多类型合并，上万只），不是 CLI 的 top-N。
  2. **自造多因子**，别只按收益排：已确立赢家(长期rank) + 动能仍在 + **反过热(剔除近3月抛物线暴涨)** + 回调不追高 + 稳健(多周期为正) + **A/C去重 + 每主题限流去集中**。
  3. 出短名单后**逐只拉证据卡验风险**（52周位置/最大回撤/RSI/夏普），再给结论——把"粗筛→精筛→诚实"三层漏斗走完，而不是止步粗筛。
  4. 收益榜单只有收益列，风险维度必须去证据卡/净值级补，别用收益排名冒充"靠谱"。

## 提交里出现个人敏感信息（2026-07-10，用户明确纠正："这是基本的"）
- 场景：仓库是 GitHub 公开仓。历史提交里出现了用户真实邮箱（HANDOFF 文档）和真实交易数据（002611 的买入金额/确认净值/App 每日收益，散布在 HANDOFF、skill-improvements、task.md、test_lots/test_reconcile、P1 spec/plan）。SMTP 授权码未入库（幸免）。
- 我做错的：写文档/测试时直接引用真实账目数字和真实邮箱当例子，没做脱敏；把"方便对账"置于"隐私"之上。
- 下次怎么避免：① 任何进 git 的内容（代码/测试/文档/commit message）不得含真实邮箱、账号、金额、持仓、收益、授权码；测试与文档一律用**虚构但算术一致**的数字和 `user@example.com` 类占位；② 真实数据只活在 `~/.quantfox`（gitignore 之外的家目录）；③ 提交前 grep 自查：邮箱模式、真实金额锚点、密码样式。

## 审查结论未经原始证据复核（2026-07-11，P2 终审）
- 场景：终审 agent 断言"universe.py 已把收益列 /100"并据此指挥了一个方向错误的修复；实际只有 fee 做了 /100，收益列是百分数单位。真实周报里"近1年 +21273.0%"暴露了积累的 100 倍显示错误。
- 我做错的：把审查者的"已核"当事实转发给修复者，没有抽查它引用的那几行代码；渲染层引入 _pct 时也没有用真实产物做一次肉眼 sanity check。
- 下次怎么避免：① 任何"已核对 X"的关键断言，主控自己 grep 那几行再采信；② 涉及单位/量纲的格式化函数上线前，用一份真实数据产物肉眼验一行；③ 单位约定（percent vs fraction）写进模块 docstring，别靠隐式记忆。
